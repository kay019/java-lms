## 1. 강의 커버 이미지 정보
- 기존에는 강의 정보 테이블에 이미지 관련 정보가 함께 저장되어 있었으나, 이미 커버 이미지용 별도 테이블이 분리되어 있어 마이그레이션 시 큰 구조 변경 없이 그대로 활용할 수 있었다.
- 따라서 이미지 저장 구조 때문에 발생할 수 있는 영향도나 스키마 충돌 없이, 신규 서비스의 도메인 모델로 매끄럽게 전환할 수 있었다.

## 3. 강사 승인 필요
> 방안2 사용 

### 방안 1 — Registration 테이블은 유지하고, ‘승인된 학생 정보 테이블’을 신규로 분리
- 승인된 학생들만을 별도로 저장하는 테이블을 생성하여, Registration은 기존처럼 “수강 신청 이력(History)” 용도로만 유지한다.
- 장점 
    - Registration 테이블은 원래 목적대로 히스토리성 테이블로 유지되므로, 데이터 정합성과 과거 기록 관리가 명확하다. 
    - 승인 로직과 조회 로직을 신규 테이블로 분리할 수 있어, 코드 구조가 더 명확해짐.

- 단점
    - 스키마가 늘어나며 테이블 간 조인 필요.
    - 신규 테이블 관리 비용 증가.

### 방안 2 — Registration 테이블에 승인 여부(boolean) 컬럼을 직접 추가
- 기존 Registration 테이블에 approved 등의 boolean 컬럼을 추가하여 승인 상태를 기록한다. 
- 장점 
    - 테이블 추가가 필요 없어 구조가 단순함.
    - 마이그레이션 복잡도가 낮고 기존 흐름을 크게 바꾸지 않아도 됨. 
- 단점 
    - 승인 여부에 따라 데이터를 분류해야 하므로 쿼리를 수정하여 Boolean 기반 필터링이 필요함.
